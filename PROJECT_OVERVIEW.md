# GymWeb — Premium Fitness Platform

> A powerful, data-driven gym management platform with a gritty, high-end aesthetic. Built as a responsive web application (PWA-style) that runs flawlessly in both mobile and desktop browsers — no native app download required.

GymWeb replaces traditional gym entry cards and manual attendance logs with **browser-based facial recognition**. Members verify their identity using their own phone camera, and the admin dashboard updates in real-time — zero hardware cost, zero API fees. The entire face-matching pipeline runs client-side via TensorFlow.js, keeping the system fast, private, and free to operate.

## 2. Database Schema Blueprint (MongoDB)

Three primary collections keep things organized. The `faceDescriptor` field in the Users collection is the key to the entire face-recognition flow.

### 2.1 Users Collection

```js
{
  firebaseId     : String,          // Firebase Auth UID
  name           : String,
  email          : String,
  role           : String,          // "user" | "admin"
  paymentStatus  : String,          // "active" | "expired" | "pending"
  isBlocked      : Boolean,
  joinDate       : Date,
  currentStreak  : Number,
  faceDescriptor : [Number]         // 128-dimensional Float32 array generated by face-api.js
}
```

> **How `faceDescriptor` works:** When a user first signs up, they take a clear baseline photo via the phone camera. `face-api.js` extracts a 128-number array that uniquely represents their face geometry. This array is saved to MongoDB and used for all future comparisons. The comparison uses **Euclidean distance** — a distance below ~0.6 = match, above = no match.

### 2.2 Attendance Collection

```js
{
  userId    : ObjectId,             // ref → Users
  timestamp : Date                  // exact check-in time
}
```

> **Tip:** Query check-ins from the last 2–3 hours to display who is currently in the gym.

### 2.3 Payments Collection

```js
{
  userId      : ObjectId,           // ref → Users
  amount      : Number,
  paymentDate : Date,
  expiryDate  : Date,
  status      : String              // "paid" | "pending" | "overdue"
}
```

## 3. Core Architecture & Workflow

### 3A. Self-Service Face Scan (User Side)

This happens **entirely on the user's mobile browser** — no admin intervention required.

```
┌─────────────────────────────────────────────────────────────────────┐
│                        USER'S PHONE BROWSER                         │
│                                                                     │
│  1. User logs in (Firebase Auth)                                    │
│  2. Taps "Verify Entry"                                             │
│  3. Browser requests camera  (navigator.mediaDevices.getUserMedia)  │
│  4. Live video feed opens                                           │
│  5. face-api.js compares live face ↔ saved faceDescriptor           │
│                                                                     │
│         ┌────────────────────┐     ┌─────────────────────┐          │
│         │  FACE MATCHES ✅   │     │  FACE MISMATCH ❌   │          │
│         └────────┬───────────┘     └─────────────────────┘          │
│                  │                                                   │
│     Check paymentStatus & isBlocked                                 │
│                  │                                                   │
│       ┌──────────┴──────────┐                                       │
│       │                     │                                       │
│  Active & Not Blocked   Blocked / Unpaid                            │
│       │                     │                                       │
│  ✅ Screen flashes       ❌ Screen flashes                          │
│  white/green             aggressive #D20A0A                         │
│  "ACCESS GRANTED"        "DENIED: Update Payment"                   │
│       │                                                             │
│  POST /api/attendance                                               │
│  (log check-in)                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**Step-by-step breakdown:**

1. **Onboarding (first time):** User signs up and takes a clear baseline photo via phone camera. `face-api.js` extracts their face descriptor (128-number array) and saves it to MongoDB.
2. **Daily Entry:** User walks into the gym, opens the website on their phone, and logs in.
3. **The Scan:** They tap "Verify Entry." The browser activates the camera and shows a live video feed.
4. **The Match:** `face-api.js` compares the live feed against the saved baseline using Euclidean distance — instantly.
5. **The Gatekeeper:**
   - **Face matches + Payment active + Not blocked →** Screen flashes white/green confirmation. API call `POST /api/attendance` logs the check-in.
   - **Blocked / Unpaid →** Screen flashes aggressive `#D20A0A` red with "DENIED: Update Payment" message.

### 3B. The Command Center (Admin Side)

The admin **does not scan anything**. They sit at a desktop or tablet and watch the data flow in.

- **Live Dashboard:** As users successfully verify their faces on their own phones, the Admin's "Live Attendees" table updates in real-time (via polling or Socket.io).
- **Control Panel:** Admin sees all users, their payment status, and a toggle to block/unblock. If they block someone, that user's next face scan attempt instantly fails.
- **Exports:** A single click triggers the backend to compile attendance + payment records into an Excel sheet and push it to the browser.

## 5. Step-by-Step Development Roadmap

### Phase 1 — The Foundation & Database

- [ ] Set up Node/Express server and connect to MongoDB.
- [ ] Create Mongoose schemas: `User` (with `faceDescriptor` field), `Attendance`, `Payments`.
- [ ] Connect Firebase Auth to the React frontend so users can register and log in securely.
- [ ] Create middleware to verify Firebase tokens on Express routes (secure all APIs).

### Phase 2 — Face-API Integration (The Hardest Part)

> This is the core engine of the entire platform. Get this right, and everything else is polish.

- [ ] Install `face-api.js` in the React project.
- [ ] Load the required neural network models into `/public`:
  - **SSD MobileNet V1** — face detection
  - **Face Landmark 68** — facial landmark detection
  - **Face Recognition Net** — descriptor extraction
- [ ] Build the camera component (`navigator.mediaDevices.getUserMedia`).
- [ ] Create the onboarding flow: capture baseline photo → extract face descriptor → save to MongoDB.
- [ ] Create the verification flow: capture live frame → extract descriptor → compare against stored descriptor using **Euclidean distance** (built into the library, threshold ~0.6).

### Phase 3 — User & Admin Dashboards

- [ ] **User UI:** Build the minimalist, dark-mode dashboard — streak, join date, and the massive "Verify Face" button.
- [ ] **Admin UI:** Use Shadcn Data Tables for the user management list.
- [ ] Wire up backend routes:
  - `PUT /api/users/:id/block` — block/unblock users
  - `GET /api/attendance` — fetch attendance logs
- [ ] Implement real-time updates for the Admin "Live Attendees" feed (polling or Socket.io).
- [ ] Implement the Excel download route using `xlsx`.

### Phase 4 — The Public Masterpiece (3D Homepage)

### Phase 5 — Final Polish (Animations & Premium UI)

## 6. Nailing the Premium Aesthetic

To make this look like a top-tier brand, adhere **strictly** to these design rules. No compromises.

| Rule            | Token / Value                      | Explanation                                                                                                              |
| --------------- | ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------ |
